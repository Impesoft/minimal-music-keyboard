### State Machine
```
NotStarted
    ↓ (LoadAsync)
Starting
    ↓ (pipe connect + reader ready)
Running
    ├→ (I/O error) → Faulted
    └→ (Dispose) → Disposed
Faulted
    └→ (Dispose) → Disposed
Disposed (terminal)
```

### Audio Thread Safety
- **Command queue:** `System.Threading.Channels.Channel<string>` (unbounded, single-reader)
  - Audio thread enqueues JSON strings without blocking or allocation
  - Background task drains channel and writes to pipe
- **Ring buffer reader:** Pre-allocated `float[] _audioWorkBuffer` in `LoadAsync`
  - `MemoryMappedViewAccessor.ReadArray<float>` call in `Read()` — no allocations
- **State reads:** `volatile bool _isReady` and `_volatile bool _disposed` for hot-path (zero-lock) reads
- **State transitions:** `object _stateLock` protects state changes

### Pipe Protocol
- Connection: `NamedPipeClientStream` client, pipe named `mmk-vst3-{bridgePid}`
- Timeout: 5 seconds (CTS-based, portable across .NET minor versions)
- Commands: JSON strings written by background task to pipe
- Audio data: MMF ring buffer (header magic: 0x4D4D4B56, validated in `Read()`)

### Disposal
- Synchronous `Dispose()` pattern (matches `SoundFontBackend`)
- Cancels pipe writer task via `_writerCts.Cancel()`
- Completes command channel (`_commandChannel.Writer.Complete()`)
- Pipe writer task drains and exits naturally (fire-and-forget)

### Error Handling
- **Partial disposal:** `NoteOffAll()` gates on `_disposed` (not `_isReady`) to enqueue final commands
- **Cancellation:** Distinguishes caller-cancel (clean rollback, no event) from internal timeout (fault + event)
- **Missing bridge:** If `mmk-vst3-bridge.exe` absent, `IsReady` stays false, warning logged, no exception thrown

## Design Justifications
1. **Why Channel<string> over ConcurrentQueue?**
   - Single-reader contract ensures queue belongs to one background task
   - ReadAllAsync() for batching without explicit drain loop
   - No allocation overhead vs. ConcurrentQueue.Enqueue

2. **Why volatile _isReady and _disposed (not _stateLock for reads)?**
   - Audio thread reads `_isReady` on every `Read()` call (million times/sec)
   - Volatile read << lock acquisition (contention-free on hot path)
   - State transitions still protected by `_stateLock` (rare operation)

3. **Why NoteOffAll gates on _disposed (not _isReady)?**
   - AudioEngine disposal order: clears `_isReady`, then disposes backends
   - NoteOffAll called during backend disposal after `_isReady` cleared
   - `_disposed` gate allows final command enqueue in this window

4. **Why Dispose doesn't await pipe writer task?**
   - Synchronous Dispose matches SoundFontBackend pattern
   - Task cancellation + channel completion signal natural exit
   - Avoids potential deadlocks from synchronous wait on background task

## Deviations from Specification (3 documented)
1. **Pipe timeout:** CTS-based 5s instead of `ConnectAsync(int, CancellationToken)` (platform-dependent availability)
2. **NoteOffAll gate:** `_disposed` instead of `_isReady` (partial disposal support)
3. **Dispose concurrency:** Non-blocking, fire-and-forget pipe writer cancellation

All deviations documented in-code and justified; behavior functionally equivalent or more correct.
