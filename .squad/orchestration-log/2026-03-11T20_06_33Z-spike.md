### Key Recommendations

1. **IInstrumentBackend abstraction** — pluggable synthesis interface. SF2 and VST3 backends both implement `ISampleProvider`.
2. **Out-of-process VST3 bridge** — native C++ executable (`mmk-vst3-bridge.exe`) hosts VST3 COM plugins. Communicates via named pipes (commands) and memory-mapped files (audio).
3. **Crash isolation** — third-party VST3 plugins crash in bridge process, not the managed host. Stability constraint preserved.
4. **Phased delivery:**
   - Phase 1: `IInstrumentBackend` + `SoundFontBackend` extraction + `AudioEngine` refactor (Medium risk)
   - Phase 2: `Vst3BridgeBackend` managed IPC client (Low risk)
   - Phase 3: `mmk-vst3-bridge.exe` native C++ (High risk)
   - Phase 4: Settings UI for VST3 configuration (Low risk)

### Rationale

- **Out-of-process over in-process COM:** Third-party native code requires isolation. Latency cost (<1ms per block) is negligible within 20ms WASAPI buffer.
- **Mixer host over separate implementations:** DRY principle — WASAPI device management, volume, enumeration stay centralized.
- **Flat catalog over grouped:** MIDI PC routing is type-agnostic. UI can group by type with LINQ.

### Risk Analysis for Gren

- Bridge lifecycle (spawn, monitor, restart, shutdown)
- Audio thread IPC (spin-wait vs semaphore on shared memory)
- SoundFontBackend extraction — regression risk on audio hot path
