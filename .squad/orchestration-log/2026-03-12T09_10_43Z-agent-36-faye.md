## Task
Diagnosed VST3 editor availability race condition where user receives "Editor not available" error despite fresh bridge build.

## Findings

**Root Cause Identified:**
- `AudioEngine.HandleVst3Instrument()` assigns `_activeBackend` to `_vst3Backend` immediately
- `LoadAsync()` runs asynchronously as fire-and-forget
- Race window: If user clicks "Open Editor" before `_isReady` becomes true, `SupportsEditor` returns false
- Result: Settings window shows "Editor not available" dialog despite bridge being present and functional

**Secondary Issue:**
- When `mmk-vst3-bridge.exe` is missing, `Vst3BridgeBackend.LoadAsync()` returns early with only `Debug.WriteLine()`
- No error event fired; backend remains assigned-but-never-ready indefinitely
- User receives no feedback about missing executable

**Bridge C++ Analysis:**
- Bridge code is correct; IPC protocol working as designed
- `Load()` properly activates buses and initializes plugin (audio_renderer.cpp:34-169)
- `OpenEditor()` validates controller and creates window (audio_renderer.cpp:423-500)
- IPC "openEditor" handler sends proper ack (bridge.cpp:107-118)
- Bug is purely on C# side

## Solution Proposed
Option A (Recommended): Defer backend assignment until load completes
- Move `Volatile.Write(_activeBackend, _vst3Backend)` into continuation after `LoadAsync()` succeeds
- Ensures `SupportsEditor` is accurate when backend becomes active
- Preserves fire-and-forget pattern; slight delay before audio plays is acceptable

**Decision forwarded to Jet (Windows Dev) for implementation.**
