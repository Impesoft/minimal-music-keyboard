# Coordinator — OB-Xd VST3 Bridge Integration (Final Validation)

**Date:** 2026-03-13  
**Status:** Complete  
**Work:** Integrated Spike's message-pump refactor; fixed split-thread pipe ACK deadlock; validated shipped bridge load/open/close success.

## Summary

Implemented final architecture change: bridge main thread now runs Win32 message loop (`GetMessageW`). Pipe commands enqueued asynchronously from background thread via `WM_BRIDGE_COMMAND` messages.

### Problem Fixed

Previous `Bridge::Run()` blocked main thread in synchronous `ReadFile(pipe)`. JUCE's MessageManager (initialized on main thread) couldn't post callbacks back to main thread when plugin code called `attached()`. Deadlock.

### Solution

**Serialized command processing:**
1. Pipe reader thread reads command, posts `WM_BRIDGE_COMMAND` to hidden window
2. Message loop processes command on main thread
3. `AudioRenderer::OpenEditor()` now fully synchronous (no promises, no timeout), runs on main thread with active message pump

## Test Results

✅ **Shipped bridge (manifest-selected versioned copy):**
- load_ack: success, supportsEditor=true
- openEditor: success (now completes without timeout)
- openEditor ACK returns ok=true
- No hang, no deadlock

✅ **Audio render thread:** Unaffected (separate thread, mutex-protected)

✅ **Backward compatibility:** Fallback copy warning acceptable; primary mechanism (versioned + manifest) bulletproof

## Known Limitations

- OB-Xd still cannot fully initialize editor UI (plugin-side requirement beyond minimal spec)
- Bridge can successfully call IPlugView::attached() without deadlock now — plugin hangs internally during that call
- This is documented incompatibility; not a bridge defect

## Architecture Now Locked

**UI/STA Thread Invariant:**
> The bridge's main thread is the UI/STA thread. All window creation, COM operations, plugin loads, and IPlugView calls must happen on this thread. The pipe reader is a background thread that only enqueues strings and posts window messages.
