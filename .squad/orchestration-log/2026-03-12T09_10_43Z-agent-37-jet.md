## Task
Fix VST3 backend race condition where `_activeBackend` assignment occurs before `LoadAsync()` completes, causing "Editor not available" errors. Also surface missing-bridge-exe failures to UI.

## Implementation

### 1. Delayed Backend Assignment (`AudioEngine.cs`)

**Changes:**
- Removed `Volatile.Write(ref _activeBackend, _vst3Backend)` from `HandleVst3Instrument()`
- Added private `async Task LoadVst3BackendAsync(InstrumentDefinition)` method
- `Volatile.Write` now executes only after `await _vst3Backend.LoadAsync()` completes **and** `_vst3Backend.IsReady == true`
- During loading, `_activeBackend` remains on previous backend (SoundFont), audio is not disrupted

**Benefit:** Eliminates race window; `SupportsEditor` is accurate when backend becomes active.

### 2. Bridge Failure Notification (`Vst3BridgeBackend.cs`)

**Changes:**
- Changed bridge-exe-missing early-return to call `BridgeFaulted?.Invoke(...)` before returning
- Existing state machine not involved (bridge never started), but subscribers are notified

**Benefit:** User receives feedback when bridge executable is missing instead of silent failure.

### 3. Load Failure Event (`IAudioEngine.cs`, `AudioEngine.cs`)

**Changes:**
- Added `event EventHandler<string>? InstrumentLoadFailed` to `IAudioEngine` interface
- `AudioEngine` constructor subscribes to `_vst3Backend.BridgeFaulted` and re-raises as `InstrumentLoadFailed`
- `SettingsWindow` subscribes on construction, unsubscribes in `ForceClose()`
- Handler dispatches to UI thread via `DispatcherQueue.TryEnqueue` and shows `ContentDialog`

**Benefit:** UI can respond to and display load failures to user.

### 4. Improved Button Feedback (`SettingsWindow.xaml.cs`)

**Changes:**
- Added third branch: when active backend is `Vst3BridgeBackend` but not yet ready
- Button shows "VST3 Plugin Still Loading" instead of generic "Editor Not Available" message

**Benefit:** Clear user feedback about loading state vs permanent unavailability.
