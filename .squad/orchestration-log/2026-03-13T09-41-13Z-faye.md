# Faye — OB-Xd Editor Diagnostics & Host Compatibility

**Date:** 2026-03-13  
**Status:** Diagnosis Complete; Final Host-Side Fix Implemented  
**Work:** IPlugFrame queryInterface fix + host-side compatibility hardening; validated against installed OB-Xd bundle.

## Summary

Previous attempt (message-pump deferral) did not resolve editor hang. Reviewed VST3 spec and HostApplication reference code; identified **IPlugFrame::queryInterface() specification violation**.

### Root Cause (Plugin Interaction)

OB-Xd likely queries `frame->queryInterface(IPlugFrame::iid)` during `attached()`. Our `EditorPlugFrame` only handled `FUnknown::iid`, not `IPlugFrame::iid`. This may cause plugin to:
- Wait indefinitely for proper frame
- Enter error recovery that deadlocks
- Attempt unsupported workarounds

### Fix Implemented

Updated `EditorPlugFrame::queryInterface()` to properly return interface when queried by `IPlugFrame::iid` (matching pattern used in HostApplication).

## Validation Against Installed OB-Xd

✅ Load succeeds: `load_ack` returns ok=true, supportsEditor=true  
✅ Controller discovery works correctly  
✅ Editor open still times out inside `attached()` after all frame/client setup  

**Conclusion:** Remaining hang is **plugin-side**, not host. OB-Xd requires features beyond minimal VST3 spec (custom run-loop, thread redesign, etc.). This is documented incompatibility.

## Host-Side Compatibility Layer

Final tweaks implemented (safe, cheap):
- Show frame window before attached()
- Show/focus client HWND before attached()
- Force redraw before attached()

## Handoff

Host-side fixes exhausted. Plugin-side hang documented. Shipped IPlugFrame fix is spec-compliant and improves compatibility for other plugins. Next: Coordinator validates end-to-end behavior.
